Getting Started with a Project

    How to structure modules, version programs, and automatically check for errors

    Design projects well

    Matters to be observed and not to be done when designing a project

    What to keep

    Make packages and hierarchies as simple as possible using them appropriately.
    Using too many hierarchies makes source code difficult to find, using too simple structures makes project distinctions difficult, and structures become too large
    Avoid storing unit tests outside the package directory
    Unit tests must be included in the sub-package of the program being developed

    The standard name for the Python installation script is setup.py
    This file is shipped with setup.cfg with installation script configuration details
    Install the package using the Python deployment utility when running setup.py
    documents stored in packages created in reStructuredText format in docs

    Prohibited

    Creating a file or module based on the functionality of the source code you want to save is not a good idea.
    Moving files back and forth for no reason does not help.

    Creating a module directory with only init.py files is unnecessary
    ex) As long as you have a hooks/init.py file, hooks.py is enough without a directory.
    When creating a directory, it is recommended that you save multiple files together that the directory means.

    The source you enter in the init.py file should be very careful
    Invoke and run this file first when recalling modules contained in the directory
    Unintentional errors cannot occur if there is incorrect content

    Version Numbering

    Software version must be marked so that it is up to date
    Users need to know how projects are developed in chronological order
    Although there is no limit to how the version number is configured, PEP 440 specifies the version format that all Python packages, applications must follow to ensure that other programs and packages easily and reliably distinguish the version of the package.
    PEP 440 defines regular expressions for version numbering

    N[.N]+[{a|b|c|rc}N][.postN][.devN]

    Standard numbering is available such as 1.2 and 1.2.3

    • 1.2 equals 1.2.0 and 1.2.3 equals 1.2.3.0

    • a (1.2a1) is an alpha version, which may be unstable and have missing features.

    • b (2.3.1b2) is a beta version that is complete but still bugged.

    • c or rc (0.4rc1) is a release candidate, a version that can be released as a final product unless a serious bug occurs.
      If both rc and c are used, the rc release is considered newer than the c release.

    • post is used for post-release releases, for minor failover, increasing the number of minor versions rather than .postN when bug fixes are released

    • dev is not recommended

      Coding Style and Auto-Check

    Introduction to standard styles and automated inspection tools

    The presence of Python-style cider named PEP 8.
    Define a standard style for creating Python code

    • Indent is 4 spaces

    • 79 characters maximum for all rows

    • Top-level functions or class definitions are separated by two blank lines

    • ASCII, encoding files using UTF-8

    • import statement imports only one module on one line, and only one module on one import statement
      Write at the top of the document, followed by comments and dogstrings
      Group by standard library imports, third-party imports, and local library imports

    • Do not use unnecessary spaces before brackets, brackets, brackets, or commas

    • Create a class name with camel notation, suffix exception error, lowercase function name and underbar
      Use leading underbar for _private properties or methods

      The existence of a pip package called a pycodestyle

      pycodestyle filename.py

      echo %ERRORLEVEL%

      Tool to catch coding errors

    • Pyflakes
      Can be installed with a pip

    • Pylint
      Check PEP 8 compliance during code error check

      PEP 8 compliance cannot be confirmed by pyflakes alone
      Need to write up to pylint to handle both code error checking and PEP 8 conformity

      Because pyflakes are faster than pylint, there is a flake8 project that allows pyflakes and pycodestyle to run at once.

      In most open source projects, flake8 is often used for code style validation.
