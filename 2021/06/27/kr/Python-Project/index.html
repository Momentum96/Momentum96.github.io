프로젝트 시작하기

  • 내용

    모듈 구조화, 프로그램 버전 부여, 자동으로 오류를 확인하는 코딩 방법

    프로젝트 잘 설계하기

    프로젝트 설계 시 지켜야 할 사항과 하지 말아야 할 사항

    지켜야 할 사항

    패키지와 계층구조를 적절히 사용하여 가능한 단순하게 만든다
    너무 많은 계층구조 사용 시 소스 코드 찾기가 힘들고 너무 단순한 구조를 사용하면 프로젝트 구분이 힘들고 구조가 비대해짐
    단위 테스트를 패키지 디렉터리가 아닌 곳에 저장하지 않도록 주의
    단위 테스트는 개발하고 있는 프로그램의 하위 패키지에 포함해야 함

    파이썬 설치 스크립트의 표준 이름은 setup.py
    이 파일은 설치 스크립트 구성 세부 정보가 포함된 setup.cfg와 함께 제공됨
    setup.py 실행 시 파이썬 배포 유틸리티를 사용해 패키지를 설치
    docs에는 reStructedText 형식으로 작성한 패키지 문서 저장

    금지 사항

    저장할 소스 코드의 기능을 기반으로 파일이나 모듈 생성하는 것은 좋지 않음
    이유없이 파일을 오가게 하는 것은 도움이 되지 않음

    init.py 파일만 있는 모듈 디렉터리를 만드는 것은 불필요한 일
    ex) hooks/init.py 파일만 있다면, 디렉터리 없이 hooks.py만으로 충분
    디렉터리 생성 시에는 디렉터리가 의미하는 여러 파일을 함께 저장하는 것이 좋음

    init.py 파일에 입력할 소스는 매우 주의해야 함
    디렉터리에 포함된 모듈을 불러올 때 가장 먼저 이 파일을 호출하고 실행
    잘못된 내용이 있으면 의도하지 않은 오류가 생길 수 없음

    버전 번호 매기기

    최신 버전임을 알 수 있도록 소프트웨어 버전을 표기해야 함
    사용자는 프로젝트가 개발되는 과정을 시간 순서대로 알아야 함
    버전 번호 구성 방법은 제한이 없으나 PEP 440은 다른 프로그램과 패키지가 패키지의 버전을 쉽고 안정적으로 구별하도록 모든 파이썬 패키지, 애플리케이션이 따라야 하는 버전 형식을 명시
    PEP 440은 버전 번호 매기기에 대한 정규표현식 정의

    N[.N]+[{a|b|c|rc}N][.postN][.devN]

    1.2, 1.2.3과 같은 표준 번호 지정 가능

    • 1.2는 1.2.0과 같고 1.2.3은 1.2.3.0과 같음

    • a(1.2a1)는 알파 버전, 불안정하고 누락된 기능이 있을 수 있는 버전

    • b(2.3.1b2)는 기능이 완료되었으나 여전히 버그가 있는 베타 버전

    • c 또는 rc(0.4rc1) 는 릴리스 후보. 심각한 버그가 발생하지 않는 한 최종 제품으로 출시될 수 있는 버전
      rc와 c는 동일 의미지만 둘 다 사용된 경우 rc 릴리스는 c 릴리스보다 새로운 것으로 간주

    • post는 출시 이후 릴리스, 사소한 오류 해결에 사용, 버그 수정 버전 출시 시에는 .postN을 쓰는 것이 아닌 마이너 버전의 번호를 늘림

    • dev는 권장하지 않음

      코딩 스타일과 자동 검사

    표준 스타일과 자동 검사 도구 소개

    PEP 8이라는 파이썬 스타일 카이드 존재
    파이썬 코드 작성을 위한 표준 스타일 정의

    • 들여쓰기는 4개의 공백

    • 모든 행은 최대 79자

    • 최상위 함수나 클래스 정의는 두개의 빈 줄로 분리

    • ASCII, UTF-8을 사용하여 파일 인코딩

    • import 문은 한 라인에 하나만, 한 import 문에서는 하나의 모듈만 가져옴
      문서 맨 위에 쓰되 주석과 독스트링 다음에 씀
      표준 라이브러리 임포트, 제3자 임포트, 로컬 라이브러리 임포트 순으로 그룹화

    • 괄호, 대괄호, 중괄호 사이 또는 쉼표 앞에 불필요한 공백 사용하지 않음

    • 클래스 이름을 낙타 표기법, 접미사 예외 오류, 소문자의 함수 이름과 언더바로 작성
      _private 속성 또는 메서드에는 선행 언더바 사용

      pycodestyle이라는 pip package 존재

      pycodestyle 파일명.py

      echo %ERRORLEVEL%

      코딩 오류를 잡는 도구

    • Pyflakes
      pip로 설치 가능

    • Pylint
      코드 오류 검사를 하는 동안 PEP 8 적합성 확인

      pyflakes만으로는 PEP 8 적합성 확인 불가
      코드 오류 검사와 PEP 8 적합성을 모두 처리하려면 pylint까지 써야함

      pylint보다 pyflakes가 좀 더 빠르기 때문에 pyflakes와 pycodestyle을 한번에 실행할 수 있게 만든 flake8 프로젝트가 있음

      대부분의 오픈소스 프로젝트에서 flake8은 코드 스타일 검증에 많이 사용함
